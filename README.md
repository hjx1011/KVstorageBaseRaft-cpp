# 《基于 C++ 与 Raft 算法的分布式强一致性 KV 存储系统学习与开发过程报告》

## 一、 项目背景与目标
在现代分布式系统中，单机存储存在“单点故障（SPOF）”的致命弱点，而简单的主从复制又难以保证数据的强一致性。为了在不可靠的网络环境中构建高可用的集群，分布式共识算法成为了核心基石。相比于晦涩难懂的 Paxos 算法，Raft 算法以其“可理解性”成为了工业界（如 etcd, TiKV）的首选。

**项目目标：**
本项目旨在从零开始，使用 C++ 手写一个基于 Raft 共识算法的分布式 KV（Key-Value）存储引擎。通过剥离复杂的工业级框架，深入底层实现 **RPC 网络通信、Raft 核心逻辑（选举与日志复制）、WAL 持久化以及快照压缩**，从而彻底掌握分布式强一致性系统的底层运行逻辑。

---

## 二、 核心技术选型
*   **开发语言**：C++ (C++11/14)，利用现代 C++ 的多线程库与面向对象特性。
*   **共识算法**：Raft 协议（完整实现 Leader 选举、日志复制、安全性限制）。
*   **网络通信**：基于 TCP Socket 手搓的轻量级 RPC 框架，支持跨进程的节点通信。
*   **存储与持久化**：WAL（Write-Ahead Logging）预写日志机制 + Snapshot 快照机制。
*   **并发控制**：多线程并发模型（`std::thread`），结合细粒度锁（`std::mutex`）与条件变量（`std::condition_variable`）保障状态机与 Raft 状态的线程安全。
*   **构建工具**：CMake。

## 三、 开发过程与核心架构演进

本项目的开发过程是一个“自底向上”构建分布式系统的过程，主要分为四个攻坚阶段：

### 第一阶段：底层通信基石 —— 轻量级 RPC 框架的实现
1.  **需求分析**：Raft 集群中的节点需要频繁交换信息（如 `RequestVote` 请求投票、`AppendEntries` 追加日志）。直接使用 HTTP 开销过大，因此需要一个高效的 RPC 框架。
2.  **核心实现**：
    *   基于底层的 TCP Socket 封装了服务端（Server）和客户端（Client）的通信模块。
    *   定义了严格的 RPC 报文格式，解决了 TCP 经典的“粘包/半包”问题。
    *   实现了请求参数的序列化与反序列化机制，使得节点间可以像调用本地函数一样，透明地发起跨进程的 Raft 协议调用。

### 第二阶段：分布式大脑 —— Raft 共识算法核心实现（最难点）
这是整个项目的灵魂所在，严格按照 Raft 论文的逻辑进行了代码落地：
1.  **Leader 选举 (Leader Election)**：
    *   实现了节点的三种状态转换：`Follower`（跟随者）、`Candidate`（候选者）、`Leader`（领导者）。
    *   **心跳与超时机制**：引入了随机化的选举超时时间（Randomized Election Timeout），有效避免了多个节点同时发起选举导致的“选票瓜分（Split Vote）”死循环。
2.  **日志复制 (Log Replication)**：
    *   客户端的写请求统一由 Leader 处理。Leader 将命令封装成日志条目（Log Entry）追加到本地，随后通过 `AppendEntries` RPC 并发地发送给所有 Follower。
    *   当收到半数以上（Majority）节点的成功响应后，Leader 将该日志标记为已提交（Committed），并应用到本地 KV 状态机，最后向客户端返回成功。
3.  **安全性保证 (Safety)**：
    *   实现了 Raft 的核心安全限制：在选举时，只有包含所有已提交日志的 Candidate 才能赢得选举（通过对比 `lastLogTerm` 和 `lastLogIndex`）。

### 第三阶段：存储优化与高可用 —— 持久化与日志快照
为了保证集群在断电或崩溃后能够恢复数据，系统必须具备持久化能力。
1.  **WAL 状态落盘**：
    *   在每次回复 RPC 之前，严格将 Raft 的核心状态（`currentTerm`、`votedFor`、`log[]`）同步写入磁盘。实现了基于文件的追加写入，确保崩溃重启后能精准恢复到宕机前的状态。
2.  **日志压缩与快照 (Log Compaction & Snapshot)**：
    *   **痛点**：随着系统运行，Raft 日志会无限增长，耗尽磁盘空间，且重启重放日志极慢。
    *   **实现**：当日志大小达到设定阈值时，触发快照机制。将当前 KV 状态机的全量数据序列化并保存为快照文件，同时丢弃快照点之前的陈旧日志。
    *   **落后节点同步**：实现了 `InstallSnapshot` RPC。当 Leader 发现某个 Follower 落后太多，且所需的日志已经被压缩掉时，Leader 会直接发送快照文件帮助其快速追赶。

### 第四阶段：业务层接入与并发优化 —— KV 状态机
1.  **KV 存储引擎**：在 Raft 共识层之上，实现了一个线程安全的内存哈希表（Hash Map），作为应用层的状态机。支持基础的 `Put`、`Get`、`Delete` 操作。
